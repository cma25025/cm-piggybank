## Phase 1 — Schema + migration + RLS + triggers (audited 2026-05-18, commit eb77d1b)

### P0 — Must fix before continuing
- `supabase/migrations/20260517120000_v1_initial.sql:801-803` — `void_transaction` uses `if v_orig is null then` to detect "not found", but `SELECT ... INTO record` with zero rows leaves the record with all NULL fields, not NULL itself. The record variable is never NULL. The `kind = 'adjustment'` guard prevents voiding the *reversing* row, but voiding the *original* parent deposit twice writes two sets of reversing adjustments and corrupts balances. Add a check: select `1 from transaction where note like 'Voided <id>%' and piggybank_id = v_orig.piggybank_id` (or, better, add a `voided_at timestamptz` column + `voided_by_transaction_id` and mark on first void). ## Phase 2 — Auth + password reset + protected routes (audited 2026-05-18, commits d24a911 + 42213e7)

### P0 — Must fix before continuing
- `src/app/signup/actions.ts:38-40` — Supabase `signUp` error message is returned verbatim to the client. When the email already exists Supabase returns `"User already registered"` (or, with confirmation off, a generic-but-distinguishable error). This propagates straight into `state.error` and the signup page renders it — full email enumeration via the signup endpoint, which defeats the carefully uniform login error at `login/actions.ts:39`. Map Supabase errors to a uniform `"Could not create account. Try signing in instead."` and log the real message server-side only. (The current `console.error` is fine for diagnostics; the return value is the leak.)
- `src/app/reset-password/actions.ts:29-36` — `resetPasswordAction` calls `updateUser({ password })` with no proof the session came from a reset link. Any authenticated user who navigates to `/reset-password` can change their own password without re-entering the current one. The page is middleware-public (PUBLIC_PATHS includes it), so a stolen laptop + an open dashboard tab = silent password rotation by the attacker. Fix: in the page (server component), call `getUser()` and check `user.recovery_sent_at`/AAL or require a fresh AMR claim; or require current-password for non-recovery sessions; or gate the page so it only renders when the request arrived via `/auth/callback?next=/reset-password` within the last N minutes (set a short-lived cookie in the callback, verify here).
- `src/middleware.ts:5-9` — `/onboarding` is not in `PUBLIC_PATHS`, but `signupAction` redirects there immediately after `signUp`. The Server Action returns a `Set-Cookie` for the new session in the same response as the 303 to `/onboarding`. The browser follows the redirect with the cookie set, middleware reads it via `request.cookies`, and the user is authenticated — so this works **iff** Supabase's `signUp` actually wrote cookies via the SSR client's `setAll` (it does, because `createClient` is awaited inside the action). Verify with one manual end-to-end run before Phase 3 lands — if the cookie write races the redirect, every new signup bounces to `/login`. Cheap to test, expensive to discover in beta.

### New findings (this re-audit)

#### P1 — Should fix before beta
- `src/app/add-money/actions.ts:90` — `p_distribution: distribution as never` — `distribution` is `{spend: number, save: number, share: number} | null`. supabase-js v2 JSON-serializes RPC params; jsonb on the Postgres side accepts the object. RPC at `migration:733-743` reads `p_distribution->>'spend'` (text extract) then `::int` casts — works for both integer JSON numbers (e.g. `1000`) and string-numbers. **Safe in practice** because `dollarsToCents` returns integers and JSON.stringify emits `1000` not `"1000"`. The `as never` cast is a Supabase types limitation (generated jsonb param type is the union `Json`), not a runtime hazard. **However**: when `distribution` is `null`, supabase-js will send `null` literal; the RPC checks `if p_distribution is not null then` (`migration:733`) — fine. Recommend swapping `as never` for `as unknown as Json` (import from generated types) so a future contributor changing the shape gets a compile error instead of `never` swallowing all type info.
- `src/app/add-money/actions.ts:86, src/app/log-spend/actions.ts:53` — no `occurred_at` field is exposed on `/add-money`. Backdating a deposit is impossible from the UI even though `add_deposit` RPC accepts no `p_occurred_at` parameter at all (`migration:695-702`) and the `transaction` table defaults to `now()`. The asymmetry with log-spend (which has full datetime-local + tz handling) is a UX hole — caretaker can backdate spend but not deposit. Either add the parameter to both, or drop from log-spend for symmetry.
- `src/app/log-spend/page.tsx:21-37` — `bucket(kind)` embed returns an array on PostgREST 1:1 embeds; the page normalizes via `Array.isArray(s.bucket) ? s.bucket[0] : s.bucket`. Works today, but if a sub is somehow missing its bucket (shouldn't happen given the schema FK + propagation triggers) `bucketKind` falls back to `"spend"` — silently mis-grouping the sub in the optgroup. Add a `console.warn` for the "shouldn't happen" branch or filter the row out.
- `src/app/activity/void-actions.ts:35` — exception parsing is `error.message.includes("already voided")`, but the RPC at `migration:144-146` raises `'void_transaction: transaction % already voided at %'` — case matches, `includes` is substring not regex, safe. **However** the other RPC failure modes (`not found`, `cannot void an adjustment`) all fall through to the generic "Couldn't void the transaction." Caretaker who triggers the rare "cannot void an adjustment" case (e.g. a manual SQL editor adjustment row appearing in their activity feed) gets a useless error. Add branches like log-spend does.
- `src/app/log-spend/log-spend-form.tsx:51` — `occurredAtISO = whenLocal ? new Date(whenLocal).toISOString() : ""` — if the user blanks the datetime-local input (some browsers allow this), `whenLocal === ""`, `occurredAtISO === ""`, the hidden field submits empty string, Zod `.optional().nullable()` (`actions.ts:13`) accepts. Server then passes `parsed.data.occurred_at || null` → RPC `p_occurred_at default null` → defaults to `now()`. Fine. But the visible input shows blank, the caretaker may think they cleared the date; round-trip is opaque. Either disallow clearing (`required`) or surface "will record as now" copy.
