## Phase 1 — Schema + migration + RLS + triggers (audited 2026-05-18, commit eb77d1b)

### P0 — Must fix before continuing
- `supabase/migrations/20260517120000_v1_initial.sql:801-803` — `void_transaction` uses `if v_orig is null then` to detect "not found", but `SELECT ... INTO record` with zero rows leaves the record with all NULL fields, not NULL itself. The record variable is never NULL. Bad UUIDs slip past the guard and the function proceeds with NULL `piggybank_id`/`kind`, then either crashes on the NOT NULL constraint or (worse) on the CHECK with a confusing error. Use `if not found then raise exception ...` immediately after the SELECT. Same bug pattern exists implicitly anywhere else that uses `record is null` — search and fix.
- `supabase/migrations/20260517120000_v1_initial.sql:789-842` — `void_transaction` is not idempotent. The `kind = 'adjustment'` guard prevents voiding the *reversing* row, but voiding the *original* parent deposit twice writes two sets of reversing adjustments and corrupts balances. Add a check: select `1 from transaction where note like 'Voided <id>%' and piggybank_id = v_orig.piggybank_id` (or, better, add a `voided_at timestamptz` column + `voided_by_transaction_id` and mark on first void). Plan §6 OV3 said "ships in Phase 5" — the data-model hook needs to land now so Phase 5 isn't a schema change.
- `supabase/migrations/20260517120000_v1_initial.sql:86-106` — no cross-table CHECK that `subcategory.bucket_id` points to a bucket whose `piggybank_id` matches `subcategory.piggybank_id`. Same gap on `transaction` (`bucket_id`/`piggybank_id`, `subcategory_id`/`piggybank_id`, `funder_id`/`piggybank_id`). A malicious or buggy insert can attach a subcategory to a bucket in another piggybank; the propagation trigger only fills `caretaker_user_id` from `piggybank_id`, so RLS still passes but the row is semantically corrupt and breaks bucket detail/activity queries. Fix with composite FKs: add `unique (id, piggybank_id)` on `bucket` and `subcategory`, then `transaction` / `subcategory` reference `(bucket_id, piggybank_id)` as a composite FK. Or accept the gap and write the application-layer validation now — the plan §6 E2/E4 implies row-level integrity, so flag explicitly.
- `supabase/migrations/20260517120000_v1_initial.sql:519-521` — comment says "UPDATE policy exists only for the balance trigger" but **no UPDATE policy is actually created on `public.transaction`**. The balance trigger updates `bucket`/`piggybank`/`subcategory` (which DO have UPDATE policies), not `transaction` itself, so the comment is just wrong — fine. But that means **`occurred_at` backfill, manual reconciliation edits, and the v1.1 edit-transaction feature are all blocked by RLS today.** Document that no UPDATE/DELETE on transaction is possible from the caretaker session (only via service-role or void). Either delete the misleading comment or add the policy now to unblock v1.1.

### P1 — Should fix before beta
- `supabase/migrations/20260517120000_v1_initial.sql:269-286` — propagation triggers on six tables share the literal name `_a_propagate_caretaker_user_id`. Postgres scopes trigger names per-table so this works, but `pg_dump` / migration replay / "drop trigger by name" tools that don't qualify by table will silently misbehave. Rename to `_a_propagate_caretaker_user_id_<table>` (e.g. `_a_propagate_caretaker_user_id_bucket`). Also note: the alphabetical-order comment at :268 is true for BEFORE triggers on the *same* table, but the `_a_` prefix only matters relative to `_b_set_created_by` and `touch_updated`. Since set_created_by doesn't read caretaker_user_id and touch_updated is BEFORE UPDATE (not INSERT), the prefix is currently decorative — fine to keep for defensive intent, but document why.
- `supabase/migrations/20260517120000_v1_initial.sql:312-326` — `touch_updated` unconditionally writes `new.updated_by_user_id := auth.uid()` even when `auth.uid()` returns NULL (service-role / cron / SQL editor). For audit-on-mutation tables, blanking the column on an admin-driven backfill is silently destructive. Either guard with `if auth.uid() is not null then` or accept NULL as "system action" and document.
- `supabase/migrations/20260517120000_v1_initial.sql:289-310` — `set_created_by` is missing from `piggybank`, `distribution_rule`, `bucket`, `request` (all have or should have `created_by_user_id` audit columns per plan §6 C8). `piggybank` and `bucket` don't have the column at all (gap vs C8 audit-cols intent). `request.submitted_by_user_id` is required so set_created_by isn't directly applicable, but the pattern should be consistent. Either add the columns and triggers or document the deliberate exclusion.
- Tests gap (plan §5 Phase 1 acceptance): only `distribution.test.ts` shipped (15 cases, in-memory only). The plan called for **six** live-DB Vitest files — `balance-invariants`, `rls`, `cascade`, `fk-ordering`, `trigger-security`, `funder-race`. None exist. This is the riskiest carry-forward into Phase 2/3: app code will be written against unverified RLS, unverified DEFERRABLE FK ordering, and unverified SECURITY DEFINER propagation. Block: needs `supabase/seed.sql` test-caretaker harness OR `supabase start` local stack + a `vitest.integration.config.ts` that points at the local Postgres. Estimate ~0.5 day for harness + ~1 day for the 6 test files. Land before Phase 4 UI starts touching real data.
- `src/lib/db/types.ts:42-65, 189-258, 307-321` — generated types still include the orphan v0 tables `accounts`, `interest_payouts`, `interest_settings`, `notes`. Commit message acknowledges these are "left in remote DB; notes has 3 rows of unknown data". App code that uses these table names will compile and reference dead schema. Drop them via a follow-up migration OR add a TypeScript lint that bans imports of these specific names. The `notes` mystery rows should be inspected before dropping (`select * from public.notes limit 5` via SQL editor); if junk, drop in next migration.
- `supabase/migrations/20260517120000_v1_initial.sql:538-555` — `v_funder_stats` joins `funder` left → `transaction` and groups by funder fields. Archived funders appear in the view with zero stats, which may or may not be intended. If the Funders screen filters `where archived_at is null` at query time, fine. If not, archived funders pollute the list. Decide and document.
- `supabase/migrations/20260517120000_v1_initial.sql:848-861` — `soft_delete_piggybank` doesn't archive the related `funder` rows or `subcategory` rows. Plan §5 doesn't specify, but consequence: after soft-delete + restore (if restore ever ships), funders/subs are still active. After hard-delete (cascade), they go. Decide now whether soft-delete is "hide" or "wind down" — affects the 30-day purge cron and the export-JSON contract (do exported funders include archived?).
- `supabase/migrations/20260517120100_drop_v0_tables.sql:8-10` — drop order is correct (`transaction_categories` first as it likely FK'd `piggybanks`), and `cascade` on the parents covers anything missed. But there's no verification step. Add a `do $$ begin if exists (select 1 from information_schema.tables where table_schema='public' and table_name in ('piggybanks','transactions','transaction_categories')) then raise exception 'v0 tables still present'; end if; end $$;` at the end as a self-test. Cheap insurance.

### P2 — Polish
- `supabase/migrations/20260517120000_v1_initial.sql:400-443` — `maintain_balances_on_transaction` UPDATE branch always subtracts-then-adds even when no balance-relevant column changed (e.g. `update transaction set note='typo fix' where id=...`). Net zero, wasteful 6 UPDATEs per no-op note edit. Optimize with `if old.kind <> new.kind or old.amount_cents <> new.amount_cents or old.bucket_id is distinct from new.bucket_id or old.subcategory_id is distinct from new.subcategory_id then ...`. Only matters once edit-transaction ships in v1.1.
- `supabase/migrations/20260517120000_v1_initial.sql:653` — possessive form `p_kid_name || '''s Piggybank'` produces "Jess's Piggybank" (correct US English) and "Alex's Piggybank" (fine). Names ending in s historically take just `'` in some style guides ("Jess' Piggybank"); APA/CMS both accept `'s`. Not a bug, just be aware that the UI cannot override the piggybank display_name during onboarding — caretaker has to edit in Settings post-create. Consider taking `p_piggybank_display_name text default null` and falling back to the possessive.
- `supabase/migrations/20260517120000_v1_initial.sql:557-570` — `v_weekly_digest` uses `date_trunc('week', occurred_at)` which is ISO week (Monday start) in Postgres. Sunday digest print (plan C7) probably wants Sunday-start weeks for kid-friendly framing. Either keep ISO and label the digest "Mon–Sun", or use `date_trunc('week', occurred_at + interval '1 day') - interval '1 day'`.
- `supabase/migrations/20260517120000_v1_initial.sql:14-16` — COPPA trigger described only in a comment. When v2 lands `owner_user_id`-as-auth-user, the rule needs to be code. - `src/lib/db/types.ts:682-685` — `signed_amount` RPC is exposed to clients via the generated types and grants. It's a pure helper; no harm, but noise in the client surface. Mark `security invoker` and consider not granting to authenticated (only the view uses it, and views inherit definer perms). Trim if not needed.
- `supabase/migrations/20260517120000_v1_initial.sql:856` — `soft_delete_piggybank` uses `update ... where id = p_piggybank_id` with no explicit `caretaker_user_id = auth.uid()` clause; RLS catches it but the error message ("not found (or not yours)") conflates two cases. Acceptable; just acknowledge the conflation is intentional (don't leak existence of others' piggybanks).
